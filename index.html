boh

The smart, simple build tool.

What is boh?

boh is a simple, smart command line build tool with plugins and colors. It's designed for fast, painless prototyping and built with simplicity in mind.

Installation

Install boh via npm.

npm install -g boh

Usage

Using boh is a simple as running boh in the directory of your project. boh will find the rules needed to build your project by itself, quickly and efficiently.

Build Rules

One of boh's core features is the ability to define build rules directly in your files. boh rules are simply YAML at the top of a file in a comment (denoted with boh). With a set of packaged rule plugins: build, link and ignore, boh is a powerful system that allows you to interface with the command line tools you know and love. Every rule thereafter is a plugin that boh will automatically require after an npm install, globally or locally.

•••
style.scss
/*boh
 * build: sass $this:../build/output.css
 * ignore: ../build/output.css
 * link: ./**/*.scss
 */
import 'login.scss';
import 'header.scss';
import 'footer.scss';

body {
    background: #f04625;
    font-family: 'Helvetica Neue', sans-serif;
    font-size: 16px;
$ boh --watch
Running boh in the current directory.

    assets/style/style.scss:build ✔ (492ms)

    src/index.js:build ✔ (497ms)

Build complete successfully.

boh is watching for changes in the current directory..

Watch

boh has the ability to watch for file changes built-in. When invoked with --watch, boh will listen for filesystem events and act on changes, discovering new rules and executing everything a normal boh run would. It does it efficiently too, boh will only execute rules based on the files that changed and not the full monty everytime.

Plugins

boh has a tiny plugin API that allow developers to define their own build rules. Plugins will be automatically required at runtime, globally and locally. All you need to do is install the plugin via npm (npm install --save-dev boh-<plugin-name>) and use it as a rule (the plugin's name corresponds to the rule name without the boh- prefix).

$ npm install --save-dev boh-uglify
boh-uglify@0.9.2 node_modules/uglify
|-- uglify-js@1.0.1

$ boh site/
Running boh in the site/ with plugins: boh-uglify ✔.

    src/index.js:uglify ✔ (567ms)

Build complete successfully.

YAML

boh rules are declared in the friendly YAML syntax. This allows for concise, extensible rule declaration right from the top of your files.

Multilingual

boh doesn't care what language you use. As long as the comment syntax of that language is supported (most are), boh will understand and build it.

Tiny

boh only needs to be installed once, globally and has just six dependencies so you don't have to download the internet every time you want to build something.

Writing build rules

boh rules are placed at the very top of the file in a comment with the boh token immediately after opening the comment. boh uses this convention for quick parsing of your files and to avoid hidden build rules deep in the source. boh supports most of the common comment syntax from the likes of Javascript or Java (// or /* */), Ruby (#) and HTML/XML (<-- -->). For an example, see the rules for this site!

Multi-line block style
/*boh
 * build: |
 *   echo "Building SASS.."
 *   sass $this:./bundle.css
 */
Single Line Comment
#boh
# build:
#   - rm -r cache/ && mkdir cache
#   - make build
#   - ./program
Inline Rule
<--boh build: mv $this ../public -->

or

//boh build: g++ $this -o program
Rule variables

boh has a helpful variable system that allows you reference the current enviornment in your rules. These variables start with a dollar ($) sign and can be escaped with a backslash (\) e.g. \$this.

$thisThe path to the current file.
$dirThe path to the directory of the current file.
Packaged Plugins

boh has three packaged plugins called build, link and ignore. These rules provide a robust build interface out of the box which gives you access to all your existing command line tools and the ability to manipulate the boh index to your liking. Rules are executed from the top down.

build {String|Array}

The build rule is one of boh's core rules. It accepts an array of commands ora multiline block string. The commands are run in sequence with the cwd being the file's directory.

/*boh
 * build:
 *   - browserify -e $this -o ../bundle.js
 *   - uglifyjs ../bundle.js -o ../bundle.min.js 
 */

link {String|Array}

The link rule links a file or directory with the current file so if boh tries to build anything in the linked directory or file, it will use the build rules from the current file. It's useful with the --watch flag so it knows what rules to run when a file changes.

/*boh
 * build: browserify -e $this -o ../bundle.js
 * link: ./**/*.js
 */

ignore {String|Array}

The ignore rule completely ignores the paths specified from the boh index. Any rules in those files will be disregarded and will not be executed by boh. Useful for when an outputted transformed file still has rules in it.

/*boh
 * build: browserify -e $this -o ../bundle.js
 * ignore: ../bundle.js
 */

Usage

$ boh --help

   boh [options] <dir>

     -h, --help        Output usage information
     -V, --version     Output the version number
     --watch           Watch for changes.
     --index           Print the boh index and exit.
     --ignore [paths]  Specify paths to ignore in the index (globs).
     -v, --verbose     Log all boh output to stdout.

Plugin Development

boh's is built on Node.js so all of it's plugins are written in Javascript. The plugin API is minimal by design. You are given the contents of your rule, the boh index and a callback. With these, users can pass your plugins parameters to configure and it's up to you to do the rest. To see some existing plugins, take a look at the packaged plugins' source.

Creating new plugins

The most important aspect of a plugin is it's name. The name of the plugin is how the user calls the from their file's header, it is the name of the rule. Plugins should be pubished on npm via with the prefix boh-. This will allow them to be automatically required and accessible when boh runs. Plugins can be installed locally or globally. Local plugins will only be discovered in the local node_modules much like how Node's require works (it will traverse up the directory tree, looking in the package.json files). Global plugins will only be required if NODE_PATH exists and points to npm's global node_modules directory. To create a new plugin, export a function that takes three parameters rule, index and callback:

Example: boh-jsfmt
var jsfmt = require('jsfmt'), fs = require('fs');

module.exports = function(rule, index, callback) {
    fs.readFile(rule.file, function(err, contents) {
        if(err) return callback(err);

        var config = jsfmt.getConfig();
        contents = jsfmt.format(contents, config);

        // Write to file, in place
        fs.writeFile(rule.file, contents, callback);
    });
}
Parameters

this{boh.Plugin} - See here for plugin methods.

rule{Object} - The currently executing rule. It has two properties: file{String}, the path to the file that the rule is found in and content{Object}, the parsed YAML content of the rule.

index{boh.Index} - The built boh index. Check out the source for a comphrensive, documented list of methods.

callback(err, data){Function} - The callback to be called when the plugin has finished with an err or data as parameters (both just pushed into the output and passed to the reporter).
